# Discord Markdown 样式指南

> 用于 AI 生成 Discord embed 内容时的格式化参考
> 
> **核心原则**：Markdown 语法不仅是格式标记，更是视觉层次和信息分组的工具

---

## 一、文本强调样式

### 1. 粗体 `**text**`
**功能**：突出关键信息、重要数值、状态标签

**适用场景**：
- 指标名称：`**成功率**: 95%`
- 状态标签：`**状态**: 正常运行`
- 重要结论：`**结论**: 系统健康`

**示例**：
```
✅ **lint 状态**: 通过（66 笔记）
📊 **总 tokens**: 125,000
⚠️ **警告**: 发现 3 个潜在问题
```

---

### 2. 斜体 `*text*` 或 `_text_`
**功能**：次要信息、补充说明、时间戳

**适用场景**：
- 时间信息：`*更新于 2026-04-19 10:00*`
- 来源标注：`*来源: [ERR-20260415-A]*`
- 补充说明：`*（可选）*`

**示例**：
```
记忆晋升报告
*日期: 2026-04-19 08:00*
*置信度: high*
```

---

### 3. 删除线 `~~text~~`
**功能**：标记已废弃、已完成、已过期的内容

**适用场景**：
- 已完成任务：`~~产品设计评审~~`
- 已修复问题：`~~Discord color 字段类型错误~~`
- 已废弃功能：`~~旧版 API（已下线）~~`

**示例**：
```
本周任务：
✅ ~~完成 MCP 适配器重构~~
🔄 优化 Discord 排版
⏳ 封装飞书日历 skill
```

---

### 4. 下划线 `__text__`
**功能**：强调行动项、待办事项、需要关注的内容

**适用场景**：
- 行动建议：`__建议立即处理__`
- 待确认项：`__需要用户确认__`
- 关键提示：`__注意：此操作不可逆__`

**示例**：
```
⚠️ 发现 10 个 weak-summary 问题
__建议本周内补充摘要__
```

---

## 二、代码块样式（信息容器）

### 1. 行内代码 `` `text` ``
**功能**：技术术语、路径、ID、配置项的视觉隔离

**适用场景**：
- 文件路径：`` `syntheses/2026-04-15-daily-log.md` ``
- 技术术语：`` `health_check` ``, `` `promotion_queue` ``
- 配置项：`` `report_type: auto` ``
- ID 标识：`` `[Wi20260419001]` ``

**示例**：
```
当前节点: `design_review`
工作项 ID: `Wi20260419001`
配置文件: `/etc/config.json`
```

---

### 2. 多行代码块 ` ```text``` `
**功能**：结构化信息容器、列表展示、表格替代

**适用场景**：
- **问题列表**（带框隔离）：
  ```
  ⚠️ 具体问题（10条）
  ```
    1. syntheses/🌟-2025-01-24-agency-problem.md
    2. syntheses/🌟-2025-01-24-llm-evaluation-fundamentals.md
    3. syntheses/🌟-2025-01-26-mutable-declarative.md
  ```
  ```

- **表格式数据**（对齐展示）：
  ```
  📊 Vault 健康
  ```
  ✅ lint 状态      通过（66 笔记，10 个 medium 问题）
  ✅ 断链数          0
  ✅ 孤立笔记        0
  ⚠️ weak-summary   10 条
  ```
  ```

- **指标面板**（紧凑展示）：
  ```
  📊 执行统计
  ```
  总笔记数: 66
  正常: 56
  异常: 10
  待处理: 11
  ```
  ```

- **日志片段**（原始输出）：
  ```
  🔍 错误详情
  ```
  [2026-04-19 10:00:00] ERROR: Connection timeout
  [2026-04-19 10:00:05] RETRY: Attempt 2/3
  [2026-04-19 10:00:10] SUCCESS: Connected
  ```
  ```

**样式特点**：
- 等宽字体，天然对齐
- 灰色背景，视觉隔离
- 适合展示结构化数据

## 三、引用样式（信息分组）

### 1. 单行引用 `> text`
**功能**：重要提示、警告信息、结论摘要

**适用场景**：
- 结论摘要：`> ✅ 系统正常，无紧急问题`
- 重要提示：`> ⚠️ 发现 3 个高优先级问题，建议立即处理`
- 操作建议：`> 💡 建议封装为独立 skill`

**示例**：
```
## 结论
> ✅ 系统正常，无紧急问题。Health/Lint/Promotion-Queue 三个接口均返回 ok=true。

## 本周建议
> 💡 建议将「飞书日历查询」封装为正式 skill，减少重复调用
```

---

### 2. 多行引用 `>>> text`
**功能**：长文本块、详细说明、多段落内容

**适用场景**：
- 详细说明：
  ```
  >>> 智元机器人在合作伙伴大会上，一口气推出六大AI基座模型（覆盖运动/交互/作业三大智能）、七大生产力解决方案，并首次公开 AIMA 全栈生态技术体系。
  
  智元同步宣布已量产下线机器人一万台，目标2026年成为"部署态元年"。
  ```

- 错误堆栈：
  ```
  >>> Traceback (most recent call last):
    File "server.py", line 42, in <module>
      result = process_data(input)
  ValueError: Invalid input format
  ```

**样式特点**：
- 左侧竖线，视觉分组
- 适合多段落内容
- 与正文明确区分

---

## 四、链接样式

### 1. 超链接 `[text](url)`
**功能**：外部资源引用、原文链接、文档跳转

**适用场景**：
- 新闻原文：`📍 [阅读原文](https://36kr.com/p/3770721219035649)`
- 文档链接：`📖 [查看文档](https://docs.example.com)`
- 相关资源：`🔗 [GitHub Issue #123](https://github.com/...)`

**示例**：
```
🔥 智元机器人发布六大AI模型
4月17日，智元机器人宣布"具身智能部署元年"开启...
📍 [阅读原文](https://36kr.com/p/3770721219035649)
```

---

### 2. 纯 URL
**功能**：直接展示完整链接（自动转为可点击）

**适用场景**：
- 需要显示完整 URL：`https://api.example.com/v1/health`
- 调试信息：`Webhook URL: https://discord.com/api/webhooks/...`

---

## 五、列表样式

### 无序列表 `- item` 或 `* item`
**功能**：并列信息、要点罗列、选项列举

**适用场景**：
- 功能列表：
  ```
  本周完成：
  - 完成 MCP 适配器重构
  - 优化 Discord 排版
  - 封装飞书日历 skill
  ```

- 问题列表：
  ```
  待处理问题：
  - 修复 Discord color 字段传递问题
  - 优化记忆晋升触发间隔
  - 封装飞书日历查询为独立 skill
  ```

**注意**：
- Discord **不支持有序列表**（`1. 2. 3.`）
- 如需编号，手动添加：`- 1. 第一项`

## 七、Embed 字段限制

### 可用 Markdown 的位置
✅ **embed.description**（4096 字符）
✅ **embed.fields[].value**（1024 字符）

### 不可用 Markdown 的位置
❌ **embed.title**（256 字符，纯文本）
❌ **embed.fields[].name**（256 字符，纯文本）
❌ **embed.footer.text**（2048 字符，纯文本）

### 字段使用建议
- **title**：用 emoji + 纯文本，如 `📰 AI 热点日报 · 周五`
- **field name**：用 emoji + 纯文本，如 `🔥 智元机器人发布六大AI模型`
- **field value**：充分使用 Markdown 格式化内容
- **description**：用于概览信息，避免过长

## 八、AI 生成内容指南

### 优先级原则
1. **视觉层次** > 严格语法：用代码块做信息容器，用引用做分组
2. **可读性** > 完整性：长文本截断，关键信息前置
3. **结构化** > 平铺：用 fields 分组，避免全挤在 description

### 内容生成流程
1. **识别信息类型**：指标/列表/表格/长文本/链接
2. **选择合适样式**：
   - 指标 → 粗体 + 行内代码
   - 列表 → 代码块（带框）
   - 表格 → 代码块（对齐）
   - 长文本 → 引用块
   - 链接 → 超链接格式
3. **组合使用**：多种样式叠加，增强视觉层次

### 示例：L5 健康巡检报告

**Description**（结论摘要）：
```
> ✅ 系统正常，无紧急问题。Health/Lint/Promotion-Queue 三个接口均返回 ok=true。
```

**Field 1: 📊 Vault 健康**（表格式）：
```
```
✅ lint 状态      通过（66 笔记，10 个 medium 问题）
✅ 断链数          0
✅ 孤立笔记        0
⚠️ weak-summary   10 条
```
```

**Field 2: ⚠️ 具体问题（10条）**（列表式）：
```
```
  1. syntheses/🌟-2025-01-24-agency-problem.md
  2. syntheses/🌟-2025-01-24-llm-evaluation-fundamentals.md
  3. syntheses/🌟-2025-01-26-mutable-declarative.md
  ...还有 7 条
```
```

**Field 3: 📊 关键指标**（粗体 + 分隔符）：
```
**总笔记数**: 66 | **待晋升**: 11 | **队列总数**: 11
```

## 九、不支持的语法

❌ **标题** `# ## ###`（embed 中无效）
❌ **有序列表** `1. 2. 3.`（不渲染）
❌ **表格** `| col1 | col2 |`（不支持）
❌ **图片** `![alt](url)`（需用 embed.image）
❌ **嵌套格式**（如 `***粗斜体***` 可能不稳定）
❌ **HTML 标签**（会被转义）

## 十、最佳实践

### ✅ 推荐做法
- 用代码块做信息容器（问题列表、表格数据）
- 用引用块突出结论和建议
- 用粗体强调关键指标
- 用行内代码隔离技术术语
- 用超链接提供原文跳转

### ❌ 避免做法
- 不要在 title/field name 中使用 Markdown
- 不要过度嵌套格式（如粗体里套斜体）
- 不要用有序列表（手动编号代替）
- 不要在代码块里再嵌套代码块
- 不要超过字段长度限制（description 4096, field value 1024）
